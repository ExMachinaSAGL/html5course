HTML5 = HTML + CSS3 + JS API

HTML5 rocks!
  • Tag semantici
    • semantica piu' espressiva (accento sulla modularità e portabilità delle pagine web, o loro parti)
    • maggiore facilità di scrittura, uso, riutilizzo
    • maggiore facilità di accesso (insieme a microdata e WAI-ARIA) per umani e sistemi automatizzati
  • API javascript
    • API "standard" per semplificare task che prima richiedevano plug-in o tecnologie proprietarie (es. Flash)
  • CSS3
    • novità utili agli sviluppatori (meno markup non semantico, script complessi, immagini...)
    • markup piu' pulito e leggibile (piu' accessibile a uomini e macchine), codice piu' manutenibile, meno risorse non necessarie (immagini), caricamento piu' veloce
    • vendor prefix: non ancora del tutto standard (ogni browser implementa il suo)
  • Mobile
    • HTML5/API/CSS3 ottimamente supportati dai browser mobile
    • alcune tecnologie introdotte sono state sviluppate proprio considerando il mobile (form fields, offline web apps, web storage)
  • Retrocompatibilità coi vecchi browser
    • cambiando il doctype le cose continuano a funzionare (la pagina continua a essere valida e visualizzata nello stesso modo)
    • IE: supporto significativo da IE9+
      • Verificare la fruibilità su browsere IE9- (fallback content, scripting, polyfills, shiv)

<time>

    

    

      
        <!-- senza @datetime deve contenere una data/time/datetime valido -->
        <time>2014-02-21</time>
        <!-- se si @datetime il contenuto puo' essere di qualsiasi tipo -->
        <time datetime="2014-02-21">il 21 di Febbraio</time>
        <!-- time -->
        <time datetime="09:00">inizieremo alle 9</time>
        <!-- datetime con time-zone (con time zone offset) -->
        <time datetime="2014-02-21T09:00+00:00">Primo incontro HTML5</time>
        <!-- datetime con "Z" (UTC) -->
        <time datetime="2014-02-21T09:00Z">Primo incontro HTML5</time>        
      

      
Macchine e uomini leggono (e interpretano) le date in maniera differente.

      

        
    
          
  • Non va utilizzato per codificare date imprecisate (es: "lo scorso inveno")
    • 
          
  • La data non puo' essere "BC": deve essere una data positiva del calendario gregoriano
    • 
          
  • L'attributo "datetime" deve essere valorizzato con una data (stringa) valida
    • 
          
  • Se l'attributo "datetime" non è specificato, deve contenere una data/time/datetime valido
    •

<meter>

    

    

      
        <meter 
          min="0" max="100" 
          low="40" high="90" 
          optimum="100" value="91">60Gb</meter>
      

      

        Spazio su disco utilizzato: 60Gb
      

      
    
      
  • Rappresenta un valore il cui range è noto: minimo (min) e massimo (max) sono definiti.
    • 
      
  • Permette di indicare quali valori vengono considerati bassi (low) e quali alti (high).
    • 
      
  • Permette di indicare quale viene considerato il valore "ideale" (optimum).
    •
